# Multi-Session MCP Server Lifecycle

## Problem

When multiple Claude Code sessions are open for the same project, the docket MCP server process (`docket.exe serve`) is a child of whichever session spawned it first. Closing that session kills the process, taking down the HTTP dashboard and checkpoint worker for all other sessions.

## Goal

Allow multiple concurrent Claude Code sessions per project, each with its own MCP server process, sharing a single dashboard instance that survives any individual session closing.

## Approach: Opportunistic Dashboard Leader

Each `docket.exe serve` instance runs MCP + checkpoint worker independently. For the shared dashboard, processes use leader election via port binding — one serves HTTP, others stand by and take over if the leader dies.

## Design

### Dashboard Lifecycle

**Startup:** When `docket.exe serve` starts, it attempts to bind the dashboard port (deterministic per-project via `portForDir()`):

- **Bound successfully:** This process is the dashboard leader. Serves HTTP as today.
- **Port taken:** This process is a standby. Starts a background goroutine that probes the port every 3 seconds. On first successful bind, starts serving.

**Tool-call guard:** Before every MCP tool handler executes, call `ensureDashboard()`:

1. Quick TCP dial to the dashboard port.
2. If connection succeeds — dashboard is alive, proceed normally.
3. If connection refused — attempt to bind the port and start serving immediately.

This guarantees the dashboard is up before any tool response is returned to Claude, eliminating the practical impact of the handoff gap.

**Shutdown:** When a process exits (stdin EOF from session close):

- If it was the leader, the port frees up. A standby detects this via its probe goroutine or on the next tool call.
- If it was the last process, everything dies naturally. No orphans.

**Port file:** `.docket/port` unchanged. All processes write the same deterministic value.

### Session-to-Feature Binding

**Problem:** SessionStart currently picks `features[0]` from `ListFeatures("in_progress")` (sorted by `updated_at DESC`) and opens a work session for it. With multiple concurrent sessions, every new session would bind to whichever feature was most recently updated. There's no mechanism to direct a specific session to a specific feature.

**Two binding paths solve this for both dashboard and manual launches.**

#### Dashboard-launched sessions: `DOCKET_LAUNCH_FEATURE` env var

The launch script is generated by docket — we control its content. Add a feature-identifying env var:

Windows `.cmd` (launch_exec_windows.go):
```batch
@echo off
set DOCKET_LAUNCH_FEATURE=my-feature-id
title docket-my-feature-id
...
claude --dangerously-skip-permissions ...
```

Unix `.sh` (launch_exec_unix.go):
```sh
#!/bin/sh
export DOCKET_LAUNCH_FEATURE=my-feature-id
cd "/project"
claude --dangerously-skip-permissions ...
```

When Claude starts from this script, it inherits `DOCKET_LAUNCH_FEATURE`. All child processes inherit it — both hooks (`docket.exe hook`) and the MCP server (`docket.exe serve`), regardless of intermediate shell wrappers.

**SessionStart hook** reads `os.Getenv("DOCKET_LAUNCH_FEATURE")`. If set, binds to that feature directly. This upgrades the correct placeholder session (matched by feature ID, not by any session-level key).

**MCP server** reads `os.Getenv("DOCKET_LAUNCH_FEATURE")` at startup. If set, knows its feature immediately.

No ambiguity: each dashboard launch produces a unique terminal with a unique env var value. Two concurrent launches for features X and Y have `DOCKET_LAUNCH_FEATURE=X` and `DOCKET_LAUNCH_FEATURE=Y` respectively.

#### Manual (organic) sessions: occupancy-aware selection

No env var is set. SessionStart picks a feature using occupancy-aware selection — skip features that already have ANY open work session (including placeholders):

```go
for _, f := range features {
    openSess, _ := s.GetOpenWorkSessionForFeature(f.ID)
    if openSess == nil {
        // Truly unoccupied — bind here
        ws, err := s.OpenWorkSession(f.ID, h.SessionID)
        // ... inject handoff context for f, return
    }
    // Reclaim zombie: Unknown session with heartbeat > 24h
    if openSess != nil && openSess.McpPid == nil &&
       openSess.ClaudeSessionID != "dashboard-launch" &&
       openSess.LastHeartbeat != nil &&
       time.Since(*openSess.LastHeartbeat) > 24*time.Hour {
        // Stale unclaimed session — treat as unoccupied
        ws, err := s.OpenWorkSession(f.ID, h.SessionID)
        // ... inject handoff context for f, return
    }
    // Has a live/recent open session (real or placeholder) — skip
}

// All features occupied — try to supersede a real (non-placeholder) session
for _, f := range features {
    openSess, _ := s.GetOpenWorkSessionForFeature(f.ID)
    if openSess != nil && openSess.ClaudeSessionID != "dashboard-launch" {
        // Real session — safe to supersede
        ws, err := s.OpenWorkSession(f.ID, h.SessionID)
        // ... inject handoff context for f, return
    }
}

// All features have placeholders or no supersedable session found
// Don't auto-bind — tell the user (bind_session instruction included)
out.SystemMessage = fmt.Sprintf(
    "[docket] Session: %s | All in-progress features are occupied.\nUse get_ready to see features, then bind_session(feature_id=\"...\", session_id=\"%s\") to pick one.",
    h.SessionID, h.SessionID,
)
```

Placeholders are treated as occupied at every stage. A manual session cannot steal a dashboard placeholder — not in the primary scan and not in the supersession fallback. Only sessions with `DOCKET_LAUNCH_FEATURE` set will match to their placeholder via `OpenWorkSession`'s placeholder upgrade path.

First manual session gets the top unoccupied feature. Second gets the next unoccupied one. If all features are occupied by real sessions, supersession occurs on the first real session. If all occupied features are placeholders, the session refuses to auto-bind rather than consuming a pending dashboard launch.

### MCP Server Session Identity

**Problem:** The MCP server needs to know which work session it belongs to for checkpoint operations and transcript access. There is no reliable out-of-band channel between hooks and the MCP server:

- Claude Code does NOT expose `CLAUDE_SESSION_ID` or any session-identifying env var to MCP server child processes (confirmed: only `CLAUDE_PLUGIN_ROOT` and `CLAUDE_PLUGIN_DATA` are documented).
- PPID matching is unreliable because SessionStart hooks run through a bash wrapper (hooks.json line 21: `bash ensure-binary.sh && docket.exe hook`), making the hook's parent `bash`, not the Claude process.
- Row-order claiming (`ClaimFirstAvailable`) is nondeterministic — two MCP servers can swap sessions.

**Solution: Hook tells Claude, Claude tells MCP server via `bind_session`.**

The only reliable path is: **SessionStart hook → system message → Claude → MCP tool call**. The hook already injects a system message. Add the `claude_session_id` to it. Claude passes it to a dedicated binding tool.

**Lifecycle rule:** Binding is the only operation that may create or resume a work session for an already-running Claude session. No other tool implicitly opens work sessions.

#### The `bind_session` tool

New MCP tool: `bind_session(feature_id, session_id)`. This is the single entry point for establishing the MCP server's identity.

**Behavior:**

1. Check if `GetWorkSessionByClaudeSession(session_id)` finds an open row → if so, cache it, set `mcp_pid`, return context.
2. Otherwise, call `OpenWorkSession(feature_id, session_id)` to create a new row (or upgrade a placeholder).
3. Set `mcp_pid = os.Getpid()` on the row.
4. Cache `ws.ID`, `ws.FeatureID`, `ws.ClaudeSessionID` in memory.
5. Return feature context (same as `get_context` output for convenience).

This handles all binding scenarios:

- **Initial bind at session start:** SessionStart opens a row, Claude calls `bind_session` → step 1 finds it.
- **Rebind after `/end-session`:** Previous row is closed. Claude calls `bind_session(feature_id="new-feature", session_id="abc123")` → step 1 finds nothing → step 2 opens a new row for the new feature.
- **Bind after "all features occupied":** SessionStart couldn't auto-bind. Claude picks a feature and calls `bind_session` → step 2 opens a row (superseding if needed).

#### Binding flows

**SessionStart hook** opens a work session (when it can) and injects:
```
[docket] Session: abc123 | Feature: my-feature (id: my-feature)
Bind docket: bind_session(feature_id="my-feature", session_id="abc123")
```

If SessionStart couldn't auto-bind (all features occupied by placeholders):
```
[docket] Session: abc123 | All features occupied.
Use get_ready to see features, then bind_session(feature_id="...", session_id="abc123") to pick one.
```

**For dashboard launches:** `DOCKET_LAUNCH_FEATURE` env var provides the feature immediately. The MCP server can claim the work session for that feature on first tool call without needing `bind_session` from Claude:
```go
featureID := os.Getenv("DOCKET_LAUNCH_FEATURE")
if featureID != "" {
    ws, _ := s.GetOpenWorkSessionForFeature(featureID)
    // Claim with mcp_pid, cache identity
}
```
`bind_session` from Claude is still accepted and acts as a no-op confirm (step 1 finds the already-claimed row).

**For manual sessions:** MCP server has no env var. It waits for `bind_session(feature_id, session_id)` to establish binding. This is deterministic — both feature_id and session_id are explicit.

**After `/end-session`:** The `/end-session` skill calls `checkpoint(end_session=true)`, which closes the work session, clears `mcp_pid`, and clears the in-memory cache. The MCP server returns to unbound mode. Claude then calls `bind_session(feature_id="next-feature", session_id="abc123")` to open a fresh row and rebind.

**Unbound mode:** Before binding, most MCP tools work fine — they operate on features, not sessions:
- Feature CRUD (add/update/delete/list/get/search) — no session identity needed
- Subtask/task/decision/issue/note operations — feature-scoped by ID parameter
- `get_ready`, `get_context`, `get_full_context` — feature-scoped

Only `checkpoint` requires session identity (for transcript access and work session closure). If called while unbound, it returns: `"No session bound. Call bind_session(feature_id=\"...\", session_id=\"...\") first — see your session context for the IDs."` Claude will self-correct.

**Why this is reliable:**
- Claude consistently follows explicit system prompt instructions, especially short ones
- The SessionStart message already exists — adding session_id and the bind_session instruction is minimal
- If Claude doesn't call bind_session immediately, all non-checkpoint tools still work; checkpoint gives a clear remediation message
- Dashboard launches bypass this entirely via env var

### Multi-Session Work Sessions

**Change `OpenWorkSession()`:** Replace the current close-all behavior (worksession.go:52-53) with feature-scoped closing:

```sql
UPDATE work_sessions SET status = 'closed', ended_at = datetime('now')
WHERE feature_id = ? AND status = 'open'
```

Opening a session for feature X closes any existing session on feature X, but leaves sessions on other features alone. This prevents two sessions from claiming the same feature while allowing concurrent work on different features.

**Supersession handling:** When session B opens on a feature that session A already has open, session A's work session row gets closed. Session A is still running, but its later Stop and SessionEnd hooks will fail to find an open work session. To handle this gracefully:

- Stop hook: if no open work session is found for this `session_id`, skip checkpointing and state transitions (already the behavior — it returns early).
- SessionEnd hook: same — if no open work session, skip handoff/checkpoint. The session was superseded; its work session data was already captured up to the point of supersession.
- This is acceptable because supersession is an explicit user action (launching a new session on the same feature). The old session's in-flight work since the last checkpoint may be lost, but this is the expected trade-off.

**Dashboard impact:** `GetActiveSessionStates()` already returns a map of feature_id to session state. With multiple concurrent sessions, this map naturally contains multiple entries. The dashboard frontend iterates this map — verify it renders correctly with multiple active features.

### Checkpoint Worker

**Per-session:** Each MCP process runs its own checkpoint worker. No changes to worker internals.

**Concurrency safety fix:** The current `DequeueCheckpointJob` implementation (checkpoint.go:120-147) uses `SELECT id ... LIMIT 1` followed by `UPDATE ... SET status='running'` inside a `db.Begin()` transaction. With SQLite's default journal mode (WAL + `DEFERRED` transactions), two workers can both SELECT the same queued row before either commits the UPDATE — both get the job.

**Fix:** Replace with a single atomic statement:

```sql
UPDATE checkpoint_jobs
SET status = 'running', started_at = datetime('now')
WHERE id = (
    SELECT id FROM checkpoint_jobs
    WHERE status = 'queued'
    ORDER BY id ASC LIMIT 1
)
RETURNING *
```

This is a single statement — SQLite guarantees atomicity at the statement level. Only one worker gets the row. If no queued jobs exist, the UPDATE affects zero rows and RETURNING yields nothing.

**Shutdown:** Worker context cancels on process exit. No orphan risk.

### Checkpoint MCP Tool Session Isolation

**Problem:** The `checkpoint` MCP tool uses `GetActiveWorkSession()` (a global query) and reads a shared `transcript-offset` file. With multiple sessions, it could checkpoint the wrong session or close someone else's work session via `end_session=true`.

**Fix:** The checkpoint handler uses the MCP server's cached identity (set during binding via `bind_session` or `DOCKET_LAUNCH_FEATURE`):

- Uses the cached work session instead of `GetActiveWorkSession()`. If unbound, returns an error directing Claude to call `bind_session(feature_id="...", session_id="...")`.
- Reads/writes transcript offset scoped by claude_session_id: `.docket/transcript-offset-{claudeSessionID}`. This is correct because transcript offsets are byte positions into per-session transcript files (`~/.claude/projects/{hash}/{sessionID}.jsonl`). Feature-scoping would be wrong — if session B supersedes session A on the same feature, A's offset is meaningless for B's transcript file.
- `end_session=true` closes only the cached work session by ID (`s.CloseWorkSession(ws.ID)`), not by global query. After closing, it **clears the in-memory binding and sets `mcp_pid = NULL`** on the row. The MCP server returns to unbound mode. The next `bind_session(feature_id, session_id)` call re-establishes binding for the new feature. This is necessary because `/end-session` is explicitly used for switching features without closing Claude — the MCP server must not keep pointing at a closed work session.
- Gets `ClaudeSessionID` from the cached work session row for `findTranscriptPath()`.

### Session-Scoped State Files

State files must be scoped to prevent cross-session interference. All files use session ID as the scope key — hooks always have it from input JSON, and the MCP server gets it from the cached work session after binding.

| Current | New | Accessed by |
|---------|-----|-------------|
| `.docket/commits.log` | `.docket/commits-{sessionID}.log` | Hooks only (PostToolUse writes, SessionEnd reads) |
| `.docket/transcript-offset` | `.docket/transcript-offset-{sessionID}` | Hooks + MCP server (both read/write via claudeSessionID) |
| `.docket/agent-nudged` | `.docket/agent-nudged-{sessionID}` | Hooks only (PreToolUse reads/writes, SessionStart clears) |
| `.docket/needs-attention-{sessionID}` | *(already scoped)* | Hooks only |
| `.docket/agent-pending-{sessionID}` | *(already scoped)* | Hooks only |

SessionStart only clears its own files. SessionEnd only reads/cleans its own files.

### Handoff File Scoping

**Problem:** SessionEnd currently rewrites handoff files for ALL active features (hook.go:381-406). It only injects fresh checkpoint data for its own feature — all other features get rewritten from historical session rows only, potentially overwriting another active session's handoff with stale content.

**Fix:** SessionEnd only writes the handoff file for its own feature (the one bound to its work session). Other active features' handoff files are left untouched — those sessions will write their own handoffs when they end.

```go
// SessionEnd: only write handoff for this session's feature
if ws != nil {
    data, err := s.GetHandoffData(ws.FeatureID)
    if err == nil {
        obs, _ := s.GetObservationsForWorkSession(ws.ID)
        mf, _ := s.GetMechanicalFactsForWorkSession(ws.ID)
        cpData := &HandoffCheckpointData{Observations: obs, MechanicalFacts: mf}
        writeHandoffFileWithCheckpoints(h.CWD, data, cpData)
    }
}
```

The `cleanStaleHandoffs()` call is also removed from SessionEnd — stale handoff cleanup should only happen when a feature is archived or deleted, not on every session close.

### Dashboard Session Liveness

**Problem:** On Windows, `isWindowAlive()` checks for a `.docket/launch/{featureID}.pid` file and returns `false` if none exists. Manually opened Claude sessions never write PID files, so the dashboard treats them as dead and closes their work sessions. A heartbeat-timeout approach breaks for idle-but-open sessions that naturally stop producing heartbeats.

**Fix:** Use the `mcp_pid` column as the primary liveness signal. The MCP server process lives exactly as long as the Claude session (it's a direct child via `.mcp.json`). Extract a shared `isSessionAlive()` helper used by both liveness call sites:

1. **`GET /api/features`** (dashboard.go:89-104) — feature list rendering.
2. **`POST /api/launch/{id}`** (dashboard.go:309-318) — launch/focus button.

```go
// SessionLiveness represents a three-valued liveness result.
type SessionLiveness int
const (
    SessionDead    SessionLiveness = iota // definitively dead — safe to close
    SessionAlive                          // definitively alive
    SessionUnknown                        // no definitive signal — don't close
)

func sessionLiveness(ws *WorkSession, projDir string) SessionLiveness {
    // 1. MCP server has claimed this session — definitive liveness check
    if ws.McpPid != nil && *ws.McpPid > 0 {
        if isPIDAlive(fmt.Sprintf("%d", *ws.McpPid)) {
            return SessionAlive
        }
        return SessionDead
    }

    // 2. Placeholder (dashboard-launched, Claude not started yet)
    if ws.ClaudeSessionID == "dashboard-launch" {
        if isWindowAlive(projDir, ws.FeatureID) {
            return SessionAlive
        }
        return SessionDead
    }

    // 3. Real session, not yet claimed by MCP server
    //    Cannot determine liveness — return Unknown
    return SessionUnknown
}
```

The dashboard uses the three-valued result:

- **`SessionDead`:** Close the work session, remove status indicator.
- **`SessionAlive`:** Show normal status badge and heartbeat.
- **`SessionUnknown`:** Show a degraded status badge (e.g., "unlinked" or dim indicator) but do NOT close the session. The session may be genuinely open — it just hasn't called a docket MCP tool yet.

Why each case is correct:

- **Claimed session (mcp_pid set):** Definitive — `isPIDAlive` checks the actual MCP server process. Works for dashboard-launched AND manual sessions. Works for idle sessions (MCP server blocks on stdio even when idle).
- **Placeholder:** Waiting for Claude to start. Check the launch PID file (existing behavior). Definitive — if the terminal process is dead, the launch failed.
- **Unclaimed real session:** SessionStart fired but no docket tool call yet. We cannot determine liveness without a process to check. Rather than guessing via heartbeat timeout (which would false-kill idle-but-open sessions), treat as unknown and leave the session open. The session will resolve to Alive once the first docket tool call sets `mcp_pid`, or to Dead when SessionEnd fires and closes it normally.

#### Zombie session reclaim

An unclaimed real session that crashes before its first `bind_session` call will stay `Unknown` forever — no `mcp_pid`, not a placeholder, SessionEnd never fired. This blocks the feature from occupancy-aware auto-binding.

**Policy:** Unknown sessions are never auto-closed during passive dashboard rendering (`GET /api/features`). They are reclaimable only on explicit action:

- **`POST /api/launch/{id}`** (dashboard launch button): If the target feature has an Unknown session with `last_heartbeat` older than 24 hours, close it and proceed with the launch. This is an explicit user action — clicking "launch" on a feature that appears stale.
- **`bind_session` tool:** If `OpenWorkSession(feature_id, session_id)` is called for a feature with an Unknown session, feature-scoped closing handles it — the stale row gets closed as part of opening the new one. This is the normal supersession path.
- **SessionStart occupancy check:** When evaluating occupancy for auto-binding, treat Unknown sessions with `last_heartbeat` older than 24 hours as unoccupied. This prevents permanent blockage from crashed sessions while still protecting recently-created sessions.

The 24-hour threshold is deliberately long. A session that has been completely idle for 24 hours without a single hook event is not actively serving a user. If it's genuinely alive, the user can always rebind explicitly.

### Dashboard Reconnection

When a dashboard leader dies and a standby takes over, browser clients experience a brief interruption. The current frontend uses polling (10-second interval via `setInterval`), not EventSource/SSE. On the next poll after the new leader starts serving, the dashboard updates normally. No data loss — all state comes from SQLite.

- **`CLAUDE.md`** (project root): The "Feature Tracking (docket)" section references `get_ready` as the entry point. Update to mention `bind_session` for explicit session binding.

These updates should be part of the implementation plan, not deferred — Claude will follow stale instructions at exactly the point where `bind_session` is required.

## Verification Tests

These tests should be written as part of implementation to validate the design:

1. **`bind_session` reopening after `checkpoint(end_session=true)`:** Open session → checkpoint with end_session=true → verify row closed and mcp_pid cleared → call bind_session with new feature → verify new row opened and mcp_pid set.
2. **24h reclaim of stale Unknown sessions:** Create an open session with no mcp_pid and heartbeat >24h ago → run SessionStart occupancy check → verify the stale session is treated as unoccupied and a new session is opened.
3. **Manual sessions refusing to consume dashboard placeholders:** Create a placeholder session for feature X → run SessionStart without DOCKET_LAUNCH_FEATURE → verify the session binds to a different feature, not X.
4. **Dashboard liveness three-valued logic:** Test all three states — Dead (mcp_pid set, process gone), Alive (mcp_pid set, process alive), Unknown (no mcp_pid, real session) — and verify the dashboard only closes Dead sessions.
